# PRD — 반도체 시료 생산주문관리 시스템

## 1. 배경 및 목적

반도체 시료의 생산 주문을 디지털로 관리하고, 시료 등록부터 생산·출고까지 전체 워크플로를 콘솔 애플리케이션으로 구현한다.

---

## 2. 메인 메뉴 구조

프로그램 시작 시 **전체 시료 요약 정보**(시료 수, 총 재고 등)를 상단에 표시한 뒤, 아래 메뉴를 출력한다. 각 메뉴 선택 시 해당 기능의 서브 메뉴(기능별 선택 화면)로 진입한다.

```
========================================
  반도체 시료 생산주문관리 시스템
  [요약] 시료: N종  총 재고: N개
========================================
 1. 시료 관리
 2. 주문 (접수 / 승인 / 거절)
 3. 모니터링
 4. 출고 처리
 5. 생산 라인
 0. 종료
```

---

## 3. 기능 요구사항

### 3.1 시료 관리

시료(Sample)는 이 시스템의 가장 기본이 되는 단위다. 각 시료는 고유한 이름과 속성을 가지며, **시스템에 등록된 시료만 주문 가능**하다.

> **수율(yield)**: 정상 시료 수 / 총 생산 시료 수. 예) 100개 생산 중 90개 정상 → 0.9

| ID | 기능 | 설명 |
|----|------|------|
| SM-01 | 시료 등록 | 이름·평균 생산시간·수율을 입력해 신규 시료 생성. ID는 자동 부여 |
| SM-02 | 시료 조회 | 등록된 모든 시료의 ID·이름·평균 생산시간·수율·재고 수량을 출력 |
| SM-03 | 시료 검색 | 이름 등 속성으로 특정 시료를 검색해 결과 출력 |

### 3.2 주문 (접수 / 승인 / 거절)

고객이 시료를 요청하면 **주문 담당자**가 주문을 생성한다.

| ID | 기능 | 설명 |
|----|------|------|
| OR-01 | 주문 접수 (시료 예약) | 주문 담당자가 시료 ID·고객명·주문 수량을 입력해 신규 주문 생성. 초기 상태 `RESERVED` |
| OR-02 | 접수 주문 목록 | `RESERVED` 상태의 주문 목록만 출력 |
| OR-03 | 주문 승인 | `RESERVED` 주문을 승인. 재고 상태에 따라 자동 분기 처리 (아래 참고) |
| OR-04 | 주문 거절 | `RESERVED` 주문을 즉시 `REJECTED`로 전이 |

#### 주문 승인 자동 분기 (OR-03)

```
주문 승인 요청
    │
    ├─ 재고 충분 (stock >= quantity)
    │       └─→ CONFIRMED  (즉시 출고 대기)
    │
    └─ 재고 부족 (stock < quantity)
            └─→ 생산 라인에 자동 등록
                    └─→ PRODUCING  (생산 진행)
```

### 3.3 모니터링

담당자가 현재 시스템 상태를 한눈에 파악할 수 있도록 주문 현황과 재고 현황을 통합 제공한다.

| ID | 기능 | 설명 |
|----|------|------|
| MO-01 | 주문량 확인 | RESERVED / CONFIRMED / PRODUCING / RELEASED 상태별 주문 목록 출력. REJECTED는 유효 주문이 아니므로 제외 |
| MO-02 | 재고량 확인 | 시료별 현재 재고 수량 및 주문 대비 재고 상태 출력 |

#### 재고 상태 기준 (MO-02)

| 상태 | 조건 |
|------|------|
| 여유 | 현재 재고가 주문 수량 대비 충분한 상태 |
| 부족 | 현재 재고가 주문 수량 대비 부족한 상태 |
| 고갈 | 재고 수량이 0인 상태 |

### 3.4 출고 처리

재고가 충분해져 `CONFIRMED` 상태가 된 주문에 대해 출고를 처리한다.

| ID | 기능 | 설명 |
|----|------|------|
| RL-01 | CONFIRMED 주문 목록 | 출고 가능한 `CONFIRMED` 상태 주문 목록 출력 |
| RL-02 | 출고 실행 | 특정 주문을 선택해 출고 처리. 주문 상태를 `RELEASED`로 전이. RELEASED 주문은 이후 삭제 불가 |

### 3.5 생산 라인

주문 승인 시 재고가 부족한 경우 생산 라인에 자동 등록된다. 수율과 오차를 고려해 실제 생산량을 산출하고, 생산 완료 시 주문 상태를 전이한다.

#### 생산량 계산 공식

| 항목 | 공식 | 비고 |
|------|------|------|
| 실 생산량 | `⌈부족분 / (수율 × 0.9)⌉` | 수율에 10% 오차를 반영해 올림 처리 |
| 총 생산 시간 | `평균 생산시간 × 실 생산량` | Sample.avgProductionTime 사용 |

> 예) 부족분 50개, 수율 0.9 → 실 생산량 = ⌈50 / (0.9 × 0.9)⌉ = ⌈61.7⌉ = **62개**

생산 완료 시 주문 상태: `PRODUCING → CONFIRMED`

#### 기능 목록

| ID | 기능 | 설명 |
|----|------|------|
| PL-01 | 생산 현황 | 현재 `PRODUCING` 상태인 주문의 정보 출력 (주문 정보, 시료명, 실 생산량, 총 생산 시간 등) |
| PL-02 | 생산 대기 큐 조회 | 생산 큐에서 대기 중인 주문 목록 출력. 스케줄링 전략: **FIFO** |

## 4. 도메인 모델

### 4.1 Sample (시료)

| 필드 | 타입 | 설명 |
|------|------|------|
| id | int | 자동 증가 식별자 |
| name | string | 시료명 |
| avgProductionTime | int | 평균 생산시간 (단위: 분) |
| yield | double | 수율 (0.0 ~ 1.0) |
| stock | int | 현재 재고 수량 |

### 4.2 Order (주문)

| 필드 | 타입 | 설명 |
|------|------|------|
| id | int | 자동 증가 식별자 |
| sampleId | int | 연관 시료 ID (등록된 시료만 허용) |
| customerName | string | 고객명 |
| quantity | int | 주문 수량 |
| status | OrderStatus | 주문 상태 |
| createdAt | string | 생성 일시 (ISO 8601) |
| updatedAt | string | 최종 수정 일시 (ISO 8601) |

### 4.3 OrderStatus 상태 전이

```
RESERVED ──→ CONFIRMED   (승인 + 재고 충분)
         ──→ PRODUCING   (승인 + 재고 부족 → 생산 라인 자동 등록)
         ──→ REJECTED    (거절)
PRODUCING ──→ CONFIRMED  (생산 완료 후 승인)
CONFIRMED ──→ RELEASED   (출고 처리)
```

유효하지 않은 전이는 `isValidTransition(from, to)` 에서 거부한다. `RELEASED` 상태의 주문은 삭제 불가.

## 6. 아키텍처 제약

- **Controller**는 콘솔 I/O를 직접 수행하지 않고 반드시 **View**에 위임
- **Repository**는 인스턴스 생성 시 JSON 파일을 로드하고, 변경 즉시 파일에 저장
- 새 `.cpp` 파일은 `SampleOrderSystem.vcxproj`에 `<ClCompile>` 항목으로 반드시 등록
