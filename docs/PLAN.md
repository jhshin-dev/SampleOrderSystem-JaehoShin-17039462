# PLAN — Phase별 개발 계획

각 Phase는 독립적으로 빌드·실행 가능한 SW를 목표로 한다.
고객님은 각 Phase 완료 후 직접 실행하여 아래 "고객님 테스트 포인트"를 중심으로 확인하고 피드백을 주시면 됩니다.

---

## Phase 1 — 기반 구조 + 시료 관리

### 개발 목표

MVC + Repository 골격을 완성하고, 시스템의 가장 기본 단위인 **시료(Sample) 관리** 기능을 동작시킨다.
이 단계에서 JSON 파일 영속성, 콘솔 메뉴 루프, 도메인 엔티티가 모두 완성된다.

### 구현 범위

| 항목 | 내용 |
|------|------|
| 아키텍처 | MVC 디렉터리 구조, `IRepository<T>`, `JsonRepository<T>` 구현 |
| 도메인 | `Sample` 구조체, `SampleRepository` |
| 기능 | SM-01 시료 등록, SM-02 시료 조회, SM-03 시료 검색 |
| 메인 메뉴 | 전체 메뉴 출력, 시료 관리(1번)만 동작 / 나머지는 "준비 중" |
| 데이터 | `data/samples.json` 자동 생성·로드 |

### 고객님 테스트 포인트

1. 프로그램 실행 시 메인 메뉴와 **시료 요약 정보**(시료 수, 총 재고)가 보이는지 확인
2. 시료를 2~3개 등록하고 프로그램을 재시작해도 **데이터가 유지**되는지 확인
3. 시료 목록에서 **이름·평균 생산시간·수율·재고**가 올바르게 표시되는지 확인
4. 이름 일부로 검색 시 **해당 시료만 필터링**되는지 확인

## Phase 2 — 주문 접수 · 승인 · 거절

### 개발 목표

고객 주문의 전체 수신 흐름을 완성한다. 승인 시 **재고 상태에 따라 CONFIRMED 또는 PRODUCING으로 자동 분기**되는 핵심 비즈니스 로직이 이 단계에서 동작한다.

### 구현 범위

| 항목 | 내용 |
|------|------|
| 도메인 | `Order` 구조체, `OrderStatus`, `OrderRepository` |
| 기능 | OR-01 주문 접수, OR-02 접수 목록, OR-03 주문 승인, OR-04 주문 거절 |
| 비즈니스 로직 | 재고 충분 → CONFIRMED, 재고 부족 → PRODUCING (생산 라인 자동 등록) |
| 데이터 | `data/orders.json` 자동 생성·로드 |

### 고객님 테스트 포인트

1. 주문 접수 시 **시료 ID·고객명·수량**을 입력하면 RESERVED 상태로 등록되는지 확인
2. **재고가 충분한 시료**로 주문 후 승인 → **즉시 CONFIRMED**로 전환되는지 확인
3. **재고가 부족한 시료**로 주문 후 승인 → **PRODUCING**으로 전환되고 "생산 라인 등록" 메시지가 보이는지 확인
4. 주문 거절 시 **REJECTED**로 전환되고 접수 목록에서 사라지는지 확인
5. 존재하지 않는 시료 ID로 주문 시 **오류 메시지**가 출력되는지 확인

## Phase 3 — 모니터링

### 개발 목표

담당자가 시스템 전체 상태를 **한눈에 파악**할 수 있는 현황판을 제공한다. 주문 상태별 분류와 시료별 재고 상태(여유·부족·고갈)를 시각적으로 확인할 수 있다.

### 구현 범위

| 항목 | 내용 |
|------|------|
| 기능 | MO-01 주문량 확인, MO-02 재고량 확인 |
| 비즈니스 로직 | REJECTED 제외, 재고 상태 3단계 판정 (여유·부족·고갈) |

### 고객님 테스트 포인트

1. 모니터링 화면에서 **RESERVED / CONFIRMED / PRODUCING / RELEASED** 상태별 주문 수가 맞는지 확인 (REJECTED는 표시되지 않아야 함)
2. 재고량 화면에서 시료별로 **여유·부족·고갈** 상태가 올바르게 표기되는지 확인
   - 재고 0 → 고갈
   - 재고 < 유효 주문량 → 부족
   - 재고 ≥ 유효 주문량 → 여유
3. Phase 2에서 만든 주문 데이터가 **모니터링에 정확히 반영**되는지 확인

---

## Phase 4 — 출고 처리

### 개발 목표

CONFIRMED 상태의 주문을 **최종 출고(RELEASED)** 처리하는 흐름을 완성한다. 주문 생애주기의 마지막 단계다.

### 구현 범위

| 항목 | 내용 |
|------|------|
| 기능 | RL-01 CONFIRMED 주문 목록, RL-02 출고 실행 |
| 비즈니스 로직 | CONFIRMED에서만 출고 가능, RELEASED 주문 삭제 불가 |

### 고객님 테스트 포인트

1. 출고 처리 화면에 **CONFIRMED 상태 주문만** 표시되는지 확인
2. 특정 주문을 선택해 출고 실행 → **RELEASED**로 전환되고 출고 목록에서 사라지는지 확인
3. CONFIRMED가 아닌 주문 ID 입력 시 **오류 메시지**가 출력되는지 확인
4. 출고된 주문이 **모니터링(RELEASED 항목)**에 반영되는지 확인

---

## Phase 5 — 생산 라인

### 개발 목표

재고 부족으로 생산 중인 주문의 **현황과 대기 큐**를 관리한다. 수율 기반 실 생산량 계산과 FIFO 스케줄링이 완성되며, 생산 완료 처리로 PRODUCING → CONFIRMED 전이가 동작한다.

### 구현 범위

| 항목 | 내용 |
|------|------|
| 기능 | PL-01 생산 현황, PL-02 생산 대기 큐 조회 |
| 비즈니스 로직 | 실 생산량 `⌈부족분 / (yield × 0.9)⌉`, FIFO 정렬, 생산 완료 시 stock 반영 후 CONFIRMED 전이 |

### 고객님 테스트 포인트

1. 생산 현황에서 PRODUCING 주문의 **실 생산량·총 생산시간**이 공식대로 계산되는지 확인
   - 예) 부족분 60, 수율 0.9 → 실 생산량 `⌈60 / 0.81⌉ = 75개`
2. 대기 큐가 **접수 순서(FIFO)** 대로 정렬되는지 확인
3. 생산 완료 처리 후 주문이 **CONFIRMED**로 전환되고, 해당 시료의 **재고가 증가**하는지 확인
4. 생산 완료 후 모니터링에서 **재고 상태가 갱신**되는지 확인
